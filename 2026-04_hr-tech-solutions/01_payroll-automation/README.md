# 🪙 Кейс: Автоматизация модуля расчета заработной платы (HR-Tech / WFM)

## 📌 Паспорт проекта
* **Бизнес-контекст**: Разработка и проектирование микросервиса расчетов выплат и компенсаций для корпоративной HRM/WFM-системы.
* **Бизнес-цель**: Исключить человеческий фактор при расчете сложных переработок, ночных смен и удержаний, обеспечив 100% соответствие алгоритмов Трудовому кодексу РФ (ТК РФ).
* **Задача аналитика**: Перевести требования законодательства (статьи ТК РФ) на технический язык алгоритмов, спроектировать ER-модель табельного учета и составить критерии приемки задач (Acceptance Criteria) для бэклога Jira.

---

## 📋 1. Постановка задачи для бэклога Jira (User Story)

### Карточка задачи: Автоматический расчет повышенной оплаты (ст. 152, 153, 154 ТК РФ)
**Как** расчетный модуль HRM-системы,  
**Я хочу** автоматически вычислять коэффициенты повышенной оплаты на основе типов отработанных часов в электронном табеле,  
**Чтобы** бухгалтерия начисляла корректные суммы без ручных перерасчетов.

#### Критерии приемки (Acceptance Criteria):
1. **Сверхурочная работа (ст. 152)**: При обнаружении в табеле за расчетный день переработки (признак `overtime`), система должна применять коэффициент **1.5x** для первых двух часов переработки и коэффициент **2.0x** для всех последующих часов в эти сутки.
2. **Работа в выходные/праздники (ст. 153)**: Если системная дата смены совпадает с официальным праздничным днем или выходным по графику сотрудника, система обязана применять коэффициент **2.0x** ко всем фактически отработанным часам за эти сутки.
3. **Ночное время (ст. 154)**: Для каждого часа с типом `night_hours` (в интервале с 22:00 до 06:00) система должна автоматически рассчитывать надбавку в размере **20%** от часовой тарифной ставки сотрудника.
4. **Приоритет пересечений**: Если ночные часы совпали со сверхурочной работой в праздничный день, коэффициенты повышенной оплаты должны суммироваться в соответствии с внутренними алгоритмами компании (начисление надбавки за ночь поверх двойного тарифа).

## 🗄️ 2. Логика удержаний и системные лимиты (ст. 137, 138 ТК РФ)

При обработке финальных расчетных листков и удержаний (по исполнительным листам, штрафам или неотработанным авансам) система должна жестко контролировать максимальные лимиты вычетов из дохода сотрудника:


(Итоговый начисленный доход) ──> (Вычитание НДФЛ) ──> (Чистый доход к выдаче)│┌─────────────────────── Валидация лимита вычетов ─────────────┘│├─> Стандартные удержания (ст. 138) ───────────────> Блокировка свыше 20%├─> По нескольким исполнительным листам ───────────> Блокировка свыше 50%└─> Алименты / Вред здоровью / Исправительные работы ─> Блокировка свыше 70%
* **Логика валидации**: При попытке провести удержание, превышающее заданный лимит для конкретной категории (`20% / 50% / 70%`), система должна автоматически уменьшать сумму списания до максимально допустимого порога, а остаток долга переносить на следующий расчетный период со статусом `deferred_debt`.

## ⚙️ 3. Логическая модель данных модуля (Табельный учет)

Для реализации алгоритма в реляционной БД спроектированы следующие ключевые сущности:
* **`Employee_Rate` (Ставки сотрудников)**: `id`, `employee_id`, `hourly_base_rate` (базовый оклад за час).
* **`TimeSheet_Record` (Записи табеля)**: `id`, `employee_id`, `work_date` (дата), `regular_hours` (норма), `overtime_hours` (сверхурочные), `night_hours` (ночные), `is_holiday` (флаг праздничного дня).
* **`Payroll_Calculation` (Итоговые начисления)**: `id`, `timesheet_record_id`, `gross_amount` (начислено), `deductions_amount` (удержано), `net_amount` (к выдаче), `status` (`Draft`, `Approved`, `Paid`).
