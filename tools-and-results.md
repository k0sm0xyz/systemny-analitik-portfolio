
## 📚 База знаний: Анализ и качество требований

В процессе работы над проектами я руководствуюсь системным подходом к валидации и очистке входящих требований перед передачей их в разработку. Ниже представлены ключевые типы выявляемых дефектов и методы их устранения.

### 1. Пять типов проблем в требованиях
*   **Противоречия** — требования, которые технически или логически невозможно выполнить одновременно (например, мгновенная отправка уведомлений vs пакетная отправка раз в час).
*   **Пробелы** — отсутствие критически важной информации для реализации (например, описан позитивный сценарий, но нет логики обработки ошибок или ролевой модели).
*   **Неоднозначности** — размытые формулировки и качественные прилагательные («быстро», «удобно», «большие объемы»). Каждое такое слово заменяется на измеримый показатель (NFR).
*   **Дублирование** — описание одного и того же требования в разных частях спецификации, что создает риск рассинхронизации при изменениях.
*   **Избыточность** — функции, не несущие бизнес-ценности для текущего проекта, которые искусственно увеличивают скоуп и бюджет разработки.

### 2. Применяемые методы анализа
1.  **Матрица трассировки требований (Traceability Matrix)** — связывание каждого функционального требования (REQ) с его источником и бизнес-целью. Помогает отсекать избыточные фичи без источника.
2.  **Анализ граничных условий** — проверка логики работы системы на краях диапазонов (отрицательные значения, пустые строки, даты из прошлого).
3.  **Сквозное тестирование сценариев** — пошаговая эмуляция действий пользователя по User Story или Use Case для поиска логических разрывов.
4.  **Прототипирование интерфейсов** — создание интерактивных макетов (Figma/Wireframes) для визуализации скрытых требований.
5.  **Командное ревью (3-Amigos)** — совместный разбор требований с лид-разработчиком и QA для раннего выявления технических нестыковок.

### 3. Артефакты фиксирования результатов
Все найденные проблемы не просто «обсуждаются», а фиксируются в Confluence / Jira в виде структурированного реестра дефектов требований:
`[ID проблемы] -> [Четкое описание] -> [Тип проблемы] -> [Предложение по исправлению/Уточняющий вопрос] -> [Статус]`
