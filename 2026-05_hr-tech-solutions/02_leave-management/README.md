# 🏖️ Кейс: Модуль «Время отдыха, Льготы & Отпуска» (HR-Tech / HRM)

## 📌 Паспорт проекта
* **Бизнес-контекст**: Проектирование функционального модуля личного кабинета сотрудника для автоматического планирования, валидации и согласования заявлений на отпуска и льготные дни отдыха.
* **Бизнес-цель**: Снизить трудозатраты кадровой службы на ручную проверку заявлений и исключить юридические риски (штрафы ГИТ) за счет автоматического контроля лимитов ТК РФ.
* **Задача аналитика**: Сформировать логику бэкенд-валидации для форм подачи заявлений, описать алгоритм расчета отпускного стажа и спроектировать правила предоставления льготных дней для уязвимых категорий сотрудников.

---

## ⚙️ 1. Функциональные требования и логика бэкенд-валидации

При отправке сотрудником электронной формы «Заявление на ежегодный оплачиваемый отпуск» система должна выполнять каскад автоматических проверок (валидаций) перед отправкой задачи на согласование руководителю.

### Критические правила валидации формы (ст. 123, 125 ТК РФ):
1. **Правило неделимой части (ст. 125)**: При планировании графика отпусков на следующий календарный год система должна проверять общую сумму дней. Если среди всех периодов сотрудника отсутствует хотя бы один непрерывный кусок длительностью **не менее 14 календарных дней**, система блокирует сохранение графика с ошибкой: `«VALIDATION_ERROR: Один из периодов отпуска обязан быть не менее 14 календарных дней»`.
2. **Проверка накопленного стажа (ст. 121, 122)**: Система разрешает авансовое взятие отпуска, но если у сотрудника текущий непрерывный стаж в компании составляет **менее 6 месяцев**, система автоматически запрашивает признак льготной категории (беременность, несовершеннолетний, многодетный). При отсутствии признака выводится предупреждение для HR-модератора.
3. **Запрет непредоставления (ст. 124)**: Система запускает ежедневный триггер-аудит базы данных. Если у сотрудника в поле `days_since_last_leave` значение превышает **640 календарных дней** (запрет неходить в отпуск 2 года подряд), система автоматически формирует критический тикет для HR-директора и блокирует возможность переноса остатков.

## 🩺 2. Модуль льгот: Социальный калькулятор для родителей (ст. 262 ТК РФ)

Для автоматизации учета дополнительных оплачиваемых выходных дней для ухода за детьми-инвалидами спроектирована специализированная логика формы.

### Логика работы калькулятора дней ухода:
* **Стандартный режим**: Система ежемесячно начисляет сотруднику **4 дополнительных оплачиваемых дня**. Неиспользованные дни сгорают в конце каждого месяца и не переносятся.
* **Продвинутый режим (Однократно в год)**: Система поддерживает бизнес-правило накопления и единовременного взятия **до 24 дней подряд** один раз в календарный год.

#### Алгоритм валидации накопленного лимита (24 дня подряд):

(Подача заявки на N дней) ──> (Проверка флага: Использован лимит в этом году?)│┌─────── Если Нет (False) ─────────┘│▼(Проверка формулы: N <= 24 AND N <= (4 * Остаток_месяцев_года))│├──> Успешно ──> Статус (Approved) ──> Списание из годового бэклога└──> Ошибка  ──> Блокировка ───────> Текст: «Превышен годовой лимит дней»

## 🗄️ 3. Расширение логической структуры БД (Сущности отпусков)

Для интеграции с общим кадровым контуром репозитория модель данных расширена следующими сущностями:
* **`Leave_Request` (Заявления на отпуск)**: `id`, `employee_id`, `start_date` (дата начала), `end_date` (дата окончания), `leave_type` (`Annual`, `Maternity`, `Unpaid`, `Child_Care_Disability`), `status` (`Draft`, `Pending_Manager`, `Approved_HR`, `Rejected`).
* **`Leave_Balance` (Баланс отпусков)**: `id`, `employee_id`, `accrued_days` (накоплено дней отпуска за текущий рабочий год), `used_days` (использовано).
* **`Employee_Benefit_Profile` (Льготный профиль)**: `id`, `employee_id`, `is_pregnant` (флаг беременности), `has_disabled_child` (наличие ребенка-инвалида), `children_count` (количество детей).
