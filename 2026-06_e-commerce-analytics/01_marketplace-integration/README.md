# 📦 Кейс: Модуль интеграции с маркетплейсами и расчета юнит-экономики (e-Commerce)

## 📌 Паспорт проекта
* **Бизнес-контекст**: Разработка интеграционного контура (OMS/ERP) для автоматизации личного кабинета селлера, торгующего на Wildberries и Ozon.
* **Бизнес-цель**: Устранить ручное обновление данных, автоматизировать синхронизацию остатков для предотвращения штрафов площадок за отмену заказов (до 3% от оборота) и внедрить автоматический расчет маржинальности товаров.
* **Задача аналитика**: Спроектировать архитектуру API-взаимодействия по схемам логистики FBS/FBO, составить функциональные спецификации для синхронизации остатков/цен и описать бизнес-требования к дашборду аналитики юнит-экономики.

---

## 🔌 1. Проектирование API-интеграции (Схемы FBS и FBO)

Система поддерживает работу по двум классическим моделям e-commerce логистики:
* **FBO (Fulfillment by Operator)**: Товар хранится на складе маркетплейса. Задача системы — раз в сутки запрашивать отчеты об остатках через API маркетплейса и обновлять внутренний аналитический контур.
* **FBS (Fulfillment by Seller)**: Товар хранится на складе селлера. Задача системы — мгновенно отправлять (push) измененные остатки в ЛК маркетплейса при любых внутренних движениях (продажи в розницу, брак, инвентаризация).

### Метод синхронизации цен и остатков: `POST /api/v1/integration/marketplace/stocks`

#### Пример структуры JSON-пакета (Запрос из внутренней ERP):
```json
{
  "seller_id": 4051,
  "marketplace": "wildberries",
  "products": [
    {
      "vendor_code": "PUMP-HYDRO-01",
      "barcode": "4601234567890",
      "stock_quantity": 42,
      "base_price": 12500.00,
      "discount_percent": 10
    }
  ]
}
```

#### Обработка ответов интеграционного шлюза:
* **`200 OK`**: Данные успешно приняты и валидированы внешней системой.
* **`422 Unprocessable Entity`**: Ошибка валидации данных (например, отрицательное количество остатков или скидка превышает допустимый бизнес-лимит в 90%).

## 📊 2. Бизнес-требования к аналитическому дашборду (Юнит-экономика)

Для категорийных менеджеров и аналитиков маркетплейсов спроектированы логические требования к экрану расчета чистой прибыли с каждой единицы товара (Unit-Economy).

### Логика расчета финансовых метрик:
Система должна автоматически вычислять итоговую маржинальность на основе агрегации данных из отчетов о реализации маркетплейсов по следующей формуле:


(Розничная цена со скидкой)│├── (Минус) Комиссия маркетплейса (зависит от категории ОКВЭД)├── (Минус) Стоимость логистики (магистраль + последняя миля + возвраты)├── (Минус) Себестоимость производства/закупки (из модуля ERP)├── (Минус) Затраты на внутреннее продвижение (Реклама ADS по API)│▼(Чистая прибыль (Net Profit)) ──> Расчет ROI и Маржинальности (%)
#### Функциональные требования к экрану:
1. **ФТ-1**: Дашборд должен автоматически подтягивать данные о расходах на внутреннюю рекламу из API Wildberries/Ozon и распределять их пропорционально количеству заказов на конкретную карточку товара.
2. **ФТ-2**: Система должна подсвечивать красным цветом позиции в таблице, у которых текущая маржинальность опустилась ниже **15%** (зона финансового риска), сигнализируя о необходимости выхода из участия в акции или пересмотра базовой цены.
