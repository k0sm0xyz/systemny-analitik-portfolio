# Кейс №1: Автоматизация бизнес-процессов кофейни

Этот кейс демонстрирует сквозное проектирование системы онлайн-предзаказа через декомпозицию требований под тремя разными углами обзора: Владельца бизнеса, Пользователя и Системного аналитика.

---

## 🪙 1. Взгляд Владельца Бизнеса (Бизнес-требования — BR)
Бизнес мыслит деньгами, конверсией, оптимизацией затрат и удержанием клиентов.

*   **BR-01 (Увеличение пропускной способности):** Сократить очереди у кассы в утренние часы пик (08:00–10:00) за счет переноса 30% заказов в онлайн.
*   **BR-02 (Рост среднего чека):** Реализовать механизм допродаж (Cross-sell) сопутствующих товаров (сиропы, выпечка) в процессе оформления заказа.
*   **BR-03 (Удержание клиентов):** Снизить отток постоянных клиентов на 15% за счет внедрения прозрачной цифровой накопительной бонусной системы вместо бумажных карточек со штампами.

---

## ☕ 2. Взгляд Клиента/Пользователя (Пользовательские требования — UR)
Пользователь мыслит личным удобством, скоростью, простотой интерфейса и выгодой.

*   **UR-01 (Заказ ко времени):** Я как *постоянный клиент* хочу оформить и оплатить заказ кофе в приложении за 10 минут до прибытия, чтобы *забрать его без очереди и не опоздать на работу*.
*   **UR-02 (Контроль готовности):** Я как *покупатель* хочу видеть актуальный статус моего заказа на экране смартфона, чтобы *не стоять у стойки выдачи в ожидании*.
*   **UR-03 (Управление бонусами):** Я как *участник программы лояльности* хочу списать накопленные баллы прямо в корзине перед оплатой, чтобы *получить скидку на текущий заказ*.

---

## 🧠 3. Самое главное: Взгляд Системного Аналитика (Системные требования — SR/API/DB)
Системный аналитик переводит бизнес-язык и юзер-стори на строгий инженерный язык архитектуры, баз данных, интеграций и граничных условий.

### ⚙️ Функциональные требования (FR) и логика бэкенда:
*   **FR-01 (Синхронизация меню):** Бэкенд приложения должен запрашивать актуальные остатки ингредиентов из кассовой системы (например, r_keeper или iiko) каждые 60 секунд. Если сироп или зерно закончились на точке, позиция в приложении должна автоматически переходить в статус `Out_of_Stock`.
*   **FR-02 (Валидация времени предзаказа):** Система должна запрещать выбор времени выдачи заказа, если до него осталось меньше минимального времени приготовления (например, < 7 минут для стандартного кофе).

### 📊 Проектирование Базы Данных (Взгляд на данные):
Для реализации программы лояльности аналитик проектирует таблицу `BONUS_ACCOUNTS`:
*   `account_id` (PK, int) — Уникальный номер счета.
*   `user_id` (FK, int) — Ссылка на пользователя.
*   `balance` (decimal) — Доступный баланс баллов (с точностью до копеек).
*   `status` (varchar) — Статус счета (`Active`, `Blocked`).

### 🔌 Интеграционный контракт (Взгляд на API):
Запрос от мобильного приложения к бэкенду на проверку корзины перед оплатой с учетом Cross-sell рекомендаций:

```json
{
  "order_init": {
    "user_id": 99421,
    "coffee_shop_id": 7,
    "items": [
      { "product_id": 101, "quantity": 1, "modifiers": ["no_sugar", "oat_milk"] }
    ]
  },
  "applied_bonuses": { "spend_points": 50 }
}
```
