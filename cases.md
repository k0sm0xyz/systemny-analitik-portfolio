# 💼 Практические кейсы (Case Studies)

## 📌 Кейс: Оптимизация процесса обработки брошенных корзин в E-commerce
*   **Роль:** Системный аналитик.
*   **Стек & Инструменты:** Jira, Confluence, Draw.io (BPMN), Postman, Figma.
*   **Методология:** Трассировка требований, анализ граничных условий, прототипирование.

### 1. Проблема и исходная задача от Бизнеса
**Сырое требование от заказчика:**
> *«Нам нужно как-то возвращать людей, которые бросили товары в корзине. Давайте система будет быстро слать им письма или пуши с напоминанием и давать хорошую скидку, чтобы они оформляли заказы. И пусть в корзине хранится много товаров, чтобы пользователь ничего не потерял».*

### 2. Выявленные дефекты требований (Анализ по методологии)
Применяя системный анализ к исходному тексту, я выявил следующие 5 типов критических проблем:

| Тип проблемы | Выявленный дефект в исходном требовании | Риск для разработки / бизнеса |
| :--- | :--- | :--- |
| **Противоречие** | «Слать письма **или** пуши» | Разработчик реализует отправку в оба канала одновременно, что вызовет спам-нагрузку и отток клиентов. |
| **Неоднозначность** | «Слать **быстро**, давать **хорошую** скидку, хранить **много** товаров» | Размытые NFR. «Быстро» для маркетинга — 5 минут, для системы — 1 мс. «Много» — нет лимита на БД. |
| **Пробел** | Описан только позитивный сценарий отправки. | Нет интеграции с CRM, не указана ролевая модель, нет логики для неавторизованных гостей. |
| **Дублирование** | Пересечение с логикой триггерных рассылок. | Риск двойной отправки промокодов из разных микросервисов. |
| **Избыточность** | «Хранить бесконечное число товаров». | Перегрузка кэша Redis/БД неактивными сессиями без явной бизнес-ценности. |

### 3. Процесс проработки и решения
1.  **Трассировка и декомпозиция (Traceability Matrix):** Привязал фичу к бизнес-цели: *«Снизить процент брошенных корзин на 15% за счет триггерного сценария»*.
2.  **Анализ граничных условий:** Ограничил время жизни брошенной корзины до 14 дней и лимит позиций в корзине до 50 товаров для защиты базы данных.
3.  **Сценарии и интеграции:** Спроектировал REST API контракт взаимодействия бэкенда интернет-магазина с сервисом рассылок (Mindbox/Sendsay).

### 4. Итоговые системные требования (Что пошло в разработку)

#### Функциональные требования (FR):
*   **FR-01:** Система должна фиксировать статус корзины как «Брошенная», если авторизованный пользователь не совершил оплату в течение 30 минут после добавления последнего товара.
*   **FR-02:** Система должна генерировать уникальный промокод на скидку 7% с ограниченным сроком действия (48 часов) через интеграцию со внутренним калькулятором скидок.
*   **FR-03:** Канал отправки выбирается каскадно: если у пользователя включены Push-уведомления в приложении — отправлять Push. Если Push недоступен — отправлять Email.

#### Нефункциональные требования (NFR / Качество):
*   **NFR-01 (Производительность):** Время выборки брошенных корзин планировщиком задач (Cron) не должно превышать 200 мс при объеме базы данных в 1 000 000 активных сессий.
*   **NFR-02 (Лимиты):** Максимальное количество уникальных SKU в корзине — не более 50 единиц. Данные неавторизованных пользователей хранятся в Redis в течение 48 часов с момента последней активности (TTL = 172800).

### 5. Результат проекта
*   Разработана прозрачная спецификация требований в Confluence, согласованная с архитекторами и QA с первого раза без доработок.
*   Реализованный триггерный сценарий снизил показатель брошенных корзин на **18.4%** за первые 3 месяца после релиза.
